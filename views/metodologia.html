Metodología

Este ranking se hace necesario porque durante los últimos años hay una explosión de portales de datos abiertos en Argentina, que los gobiernos utilizan para transparentar su gestión. Publican datos desagregados que registran el accionar del estado, utilizando licencias y formatos abiertos que facilitan la reutilización.

Pero no todos los portales alcanzan los mismos niveles calidad. Para promover buenas prácticas y destacar los portales de mayor calidad en el país, incluyímos portales de ciudades, provincias, ministerios y otras instituciones públicas.

Un portal de datos debe ofrecer datos de calidad, confiables y que sean fácilemente reutilizables. Para poder evaluar si lo logran, definimos los siguientes criterios. Hay criterios de evaluación automática y otros de evaluación manual, algunos cubren el portal en general y otros se calculan por cada conjunto de datos y los recursos que este contiene.

Criterios por portal

Estos criterios aplican a todo el portal en general. El valor de cada criterio promedia para crear el puntaje parcial para cada grupo de criterios.

Criterios de facilidad de navegación del portal

Uso de identidad oficial (manual)

Para aumentar la confiabilidad de la información publicada, la entidad responsable del portal debe ser fácil de identificar, es necesario utilizar logo o isotipo de la entidad (municipio, ministerio, institución) en todas las páginas.

Valores: 1 si cumple, 0 si no cumple.

Enlace al sitio oficial de la entidad (manual)

Para facilitar la verificación de que es un portar oficial, el sitio web de la entidad responsable del portal debe ser fácil de acceder, es necesario colocar un enlace al sitio web de la entidad (municipio, ministerio, institución) en todas las páginas.

Valores: 1 si cumple, 0 si no cumple.

Explicación de datos abiertos (manual)

Para muchos ciudadanos, el concepto de datos abiertos es novedoso, y por eso es necesario incluir una descripción de este concepto en la portada, o alternativamente en una página dedicada.

Valores: 1 si cumple, 0 si no cumple.

Enlace a todos los conjuntos de datos (manual)

Para facilitar la comprensión de todo el contenido del portal, el listado completo de todos los conjuntos de datos debe poder ser accesible.

Valores: 1 si cumple, 0 si no cumple.

Enlace a todos los conjuntos de datos en portada (manual)

Para faciliar la comprensión de todo el contenido del portal desde la primera visita, el enlace a todos los conjuntos de datos debe poder ser accesible desde la portada.

Valores: 1 si cumple, 0 si no cumple.

Sistema de búsqueda de conjunto de datos (manual)

Para facilitar la búsqueda de conjuntos de datos y recursos en el portal, debe existir un sistema de búsqueda que permita utlizar nombres, etiquetas y contenido para encontrar conjuntos de datos y devolver resultados relevantes.

Valores: 1 si cumple, 0 si no cumple.

Accesibilidad web (automático)

Para garantizar que el sitio puede ser precibido, operado y comprendido por la mayor diversidad de usuarios y dispositivos, el sitio web debe ser accesible, se utilizará la tecnología de Examinator.ws para realizar una evaluación automatizada (parcial) del cumplimiento con las normativas WAI-WCAG2, cuyo cumplimiento es exigido por la ley nacional 26.653.

Valores: entre 0 y 1, según la metodología de Examinator.ws.

Criterios de facilidad de uso automatizado de los datos

Existencia de API (manual)

Para facilitar el acceso automatizado a los conjuntos de datos se requiere una interfaz especial para que los sistemas accedan a los datos. El portal debe tener una API REST. La API puede requerir el uso de un Token, siempre y cuando no sea necesario registrarse para obtenerlo.

Valores: 1 si cumple, 0 si no cumple.

Documentación de API (manual)

Para que los desarrolladores de software puedan fácilmente desarrollar sistemas que consuman automáticamente datos del portal de datos, es necesario un documento que detalle el funcionamiento de la API REST.

Valores: 1 si cumple, 0 si no cumple.
Criterios por conjunto de datos

Estos criterios aplican a cada conjunto de datos y sus recursos, y sus valores son promediados para calcular el puntaje general del portal.

Criterios de calidad de metadatos

Responsable (automático)

Los metadatos del conjunto de datos deben expresar un responsable para los mismos, esto incrementa la confiabilidad y facilita la retroalimentación.

Valores: 1 si cumple, 0 si no cumple.

Frecuencia de actualización declarada (automático)

Los metadatos del conjunto de datos deben definir una expectativa para la actualización de los datos, esto aumenta la confiabilidad y permite planificar análisis periódicos.

Valores: 1 si cumple, 0 si no cumple.

Frecuencia de actualización efectiva (automático)

El tiempo transcurrido desde la última fecha de actualización no debe ser superior a la frecuencia de actualización declarado. Los datos actualizados permiten realizar análisis con información más reciente.

Licencia (automático)

Los metadatos del conjunto de datos deben expresar los términos en los que este se comparte, habilitando el uso para cualquier fin y requiriendo, como máximo, la atribución. Esto evita dudas sobre la necesidad de pedir autorización para la reutilización, facilitándola.

Ver listado de licencias libres (en construcción).

Para sugerir otras licencias, Contáctenos.

Valores: 1 si cumple, 0 si no cumple.

Formato (automático)

La extensión de archivo del recurso principal del conjunto de datos debe ser la de un formato cuya interpretación o reproducción no esté limitada por patentes o regalías. Esto elimina barreras para la reutilización.

Ver listado de formatos abiertos (en construcción).

Para sugerir otros formatos, Contáctenos.

Valores: 1 si cumple, 0 si no cumple.

Criterios de calidad de datos

Validez

Los recursos de cada conjunto de datos que puedan ser validados serán evaluados en su adherencia a los requisitos del formato, lo que facilita su reutilización.

Para cada uno de los formatos validados se utiliza una herramienta diferente:

  • CSV: Goodtables
  • JSON: JSON Lint (no implementado)
  • XML: (no implementado)
  • KML: (no implementado)
  • GeoJSON: (no implementado)

Para sugerir otros formatos o validadores, Contáctenos.

Valores: 1 si cumple, 0 si no cumple.

Historial de versiones

La metodología de evaluación fué creada durante el año 2016 por la Fundación Concimiento Abierto, basado en la metodología del Open Data Index, criterios de usabilidad y nuestra propia experiencia en uso y publicación de portales de datos.

Sin embargo esta está en actualización y revisión constante.

  • Versión 0.1: 09/02/2017 - Versión incial basada en metodología OpenDataIndex y criterios de usabilidad.

Por consultas sobre nuestra metodología, por favor contáctese. Adicionalmente, durante el período de versión beta, alentamos a enviar sugerencias de mejoras.
