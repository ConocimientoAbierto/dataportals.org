Metodología

Este ranking se hace necesario porque durante los últimos años hay una explosión de portales de datos abiertos en Argentina, que los gobiernos utilizan para transparentar su gestión. Publican datos desagregados que registran el accionar del estado, utilizando licencias y formatos abiertos que facilitan la reutilización.

Para promover buenas prácticas y destacar los portales de mayor calidad en el país, incluímos portales de ciudades, provincias, ministerios y otras instituciones públicas.

Un portal de datos debe ofrecer datos de calidad, confiables y que sean fácilmente reutilizables.

Para poder evaluar si lo logran, definimos los siguientes criterios. Hay criterios de evaluación automática y otros de evaluación manual, algunos cubren el portal en general y otros se calculan por cada conjunto de datos y los recursos que este contiene.

Criterios por portal

Estos criterios aplican a todo el portal en general. El valor de cada criterio promedia para crear el puntaje parcial para cada grupo de criterios.

Criterios de facilidad de navegación del portal

Uso de identidad oficial (manual)

Para aumentar la confiabilidad de la información publicada, la entidad responsable del portal debe ser fácil de identificar, es necesario utilizar logo o isotipo de la entidad (municipio, ministerio, institución) en todas las páginas.

Valores: 1 si cumple, 0 si no cumple.

Enlace al sitio oficial de la entidad (manual)

Para facilitar la verificación de la relación del portal con la entidad, el sitio web de la entidad responsable del portal debe ser fácil de acceder. Es necesario colocar un enlace al sitio web de la entidad (municipio, ministerio, institución) en todas las páginas.

Valores: 1 si cumple, 0 si no cumple.

Explicación de datos abiertos (manual)

Para muchos ciudadanos, el concepto de datos abiertos es novedoso, y por eso es necesario incluir una descripción de este concepto en la portada, o alternativamente en una página dedicada. El texto debe tener más de 20 palabras, mencionar algunos de los beneficios de los datos abiertos.

Valores: 1 si cumple, 0 si no cumple.

Enlace a todos los conjuntos de datos (manual)

Para facilitar la comprensión de todo el contenido del portal, el listado completo de todos los conjuntos de datos debe poder ser accedido. Esto es para evitar listas por categoría que pueden resultar incompletas.

Valores: 1 si cumple, 0 si no cumple.

Sistema de búsqueda de conjunto de datos (manual)

Para facilitar la búsqueda de conjuntos de datos y recursos en el portal, debe existir un sistema de búsqueda que permita utilizar nombres, etiquetas y contenido para encontrar conjuntos de datos y devolver resultados relevantes.

Valores: 1 si cumple, 0 si no cumple.

Criterios de facilidad de uso automatizado de los datos

Existencia de API (manual)

Para facilitar el acceso automatizado a los conjuntos de datos se requiere una interfaz especial para que los sistemas accedan a los datos. El portal debe tener una API REST. La API puede requerir el uso de un Token, siempre y cuando no sea necesario registrarse para obtenerlo.

Valores: 1 si cumple, 0 si no cumple.

Documentación de API (manual)

Para que los desarrolladores de software puedan fácilmente desarrollar sistemas que consuman datos del portal, es necesario un documento que detalle el funcionamiento de la API REST.

Valores: 1 si cumple, 0 si no cumple.
Criterios por conjunto de datos

Estos criterios aplican a cada conjunto de datos y sus recursos, y sus valores son promediados para calcular el puntaje general del portal.

Criterios de calidad de metadatos

Explicación de los datasets (automática)

Cada conjunto de datos debe tener una descripción que explique el contenido de los campos, acrónimos, abreviaciones y cualquier otra aclaración necesaria para comprender el sentido de los datos y facilitar su reutilización.

La descripción debe estar presente como campo de metadatos y tener una longitud superior a los 150 caracteres.

Valores: 1 si cumple, 0 si no cumple.

Responsable (automático)

Los metadatos del conjunto de datos deben expresar quién es el responsable del mismo, esto incrementa la confiabilidad y facilita la retroalimentación.

La dirección de correo electrónico del responsable debe estar publicada.

Valores: 1 si incluye email, 0.5 si está presente, 0 si no está presente.

Frecuencia de actualización declarada (automático)

Una expectativa para la actualización de los datos debe estar definida en metadatos del conjunto de datos, esto aumenta la confiabilidad y permite planificar análisis periódicos.

La frecuencia de actualización no debe decir "a demanda", "eventual" o ser inespecífica en algún otro sentido.

Valores: 1 si cumple, 0 si no cumple.

Frecuencia de actualización efectiva (automático)

El tiempo transcurrido desde la última fecha de actualización no debe ser superior a la frecuencia de actualización declarado. Los datos actualizados permiten realizar análisis con información más reciente.

Valores: 1 si cumple, 0 si no cumple.

Licencia (automático)

Los términos de la licencia con la que se comparte un dataset deben estar expresados en los metadatos de cada conjunto de datos. La licencia debe habilitar el uso para cualquier fin y requiriendo, como máximo, la atribución. Esto evita dudas sobre la necesidad de pedir autorización para la reutilización.

Ver listado de licencias libres (en construcción).

Para sugerir otras licencias, Contáctenos.

Valores: 1 si cumple, 0 si no cumple.

Formato (automático)

El conjunto de datos debe contener al menos un recurso cuya extensión de archivo corresponda con la de un formato abierto. Los formatos abiertos son aquellos cuya interpretación o reproducción no está limitada por patentes o regalías. Esto elimina barreras para la reutilización.

Ver listado de formatos abiertos (en construcción).

Para sugerir otros formatos, Contáctenos.

Valores: 1 si cumple, 0 si no cumple.

Criterios de calidad de datos

Validez

Los recursos de cada conjunto de datos que puedan ser validados serán evaluados en su adherencia a los requisitos del formato, lo que facilita su reutilización.

Para cada uno de los formatos validados se utiliza una herramienta diferente:

  • CSV: Goodtables
  • JSON: JSON Lint (no implementado)
  • XML: (no implementado)
  • KML: (no implementado)
  • GeoJSON: (no implementado)

Para sugerir otros formatos o validadores, Contáctenos.

Valores: 1 si cumple, 0 si no cumple.

Historial de versiones

La metodología de evaluación fue creada durante el año 2016 por la Fundación Conocimiento Abierto, basado en la metodología del Open Data Index, criterios de usabilidad y nuestra propia experiencia en uso y publicación de portales de datos.

Sin embargo esta está en actualización y revisión constante.

  • Versión 0.1: 09/02/2017 - Versión inicial basada en metodología OpenDataIndex y criterios de usabilidad.

Por consultas sobre nuestra metodología, por favor contáctese. Adicionalmente, durante el período de versión beta, alentamos a enviar sugerencias de mejoras.
