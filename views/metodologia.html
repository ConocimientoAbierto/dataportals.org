Metodología

Durante los últimos años hay una explosión de portales de datos abiertos en argentina, pero algunos de estos varían en grado de calidad. Para promover buenas prácticas y destacar los portales de mayor calidad en el país, incluyendo portales de ciudades, provincias, ministerios y otras instituciones.

Un portal de datos debe ofrecer datos de calidad, confiables y facilitar la reutilización, para poder evaluarlos definimos los siguientes criterios, algunos son de evaluación automática y otros de evaluación manual. Algunos cubren el portal en general y otros se calculan por conjunto de datos.

Criterios por portal

Estos criterios aplican a todo el portal en general

Criterios de facilidad de uso del portal

Uso de identidad oficial

La entidad responsable del sitio web debe ser fácil de identificar, es necesario utilizar logo o isotipo de la entidad (municipio, ministerio, institución) en todas las páginas.

Enlace al sitio oficial de la entidad

La entidad responsable del sitio web debe ser fácil de acceder, es necesario colocar un enlace al sitio web de la entidad (municipio, ministerio, institución) en todas las páginas.

Explicación de datos abiertos

Para muchos ciudadanos, el concepto de datos abiertos es novedoso, y por eso es necesario incluir una descripción de este concepto en la portada y/o página dedicada.

Enlace a todos los conjuntos de datos

El listado completo de todos los conjuntos de datos debe poder ser accesible.

Enlace a todos los conjuntos de datos en portada

El listado completo de todos los conjuntos de datos debe poder ser accesible desde la portada.

Sistema de búsqueda de conjunto de datos

Debe ser fácil de encontrar un conjunto de datos mediante el sistema de búsqueda.

Examinator.ws (automático)

El sitio web debe ser accesible, se utilizará la tecnología de examinator para realizar una evaluación automatizada (parcial) del cumplimiento con las normativas WAI-WCAG2.

Criterios de facilidad de uso de los datos

Existencia de API

Para facilitar el acceso automatizado a los conjuntos de datos se requiere una interfaz especial para que los sistemas accedan a los datos.

Documentación de API

Para que los desarrolladores de software puedan fácilmente desarrollar sistemas que consuman datos del portal de datos, es necesario un documento que detalle el funcionamiento de la API.

Criterios por conjunto de datos

Estos criterios aplican a cada conjunto de datos y sus recursos

Criterios de facilidad de uso de los datos

Explicación de los datasets (automático - manual)

Cada conjunto de datos debe tener una descripción que explique el contenido de los campos, acrónimos, abreviaciones y cualquier otra aclaración necesaria para comprender el sentido de los datos y facilitar su reutilización.

Criterios de calidad de metadatos

Responsable (automático)

Los metadatos del conjunto de datos deben expresar un responsable para los mismos, esto incrementa la confiabilidad y facilita la retroalimentación.

Frecuencia de actualización declarada (automático)

Los metadatos del conjunto de datos deben definir una expectativa para la actualización de los datos.

Frecuencia de actualización efectiva (automático)

El tiempo transcurrido desde la última fecha de actualización no debe ser superior a la frecuencia de actualización declarado.

Licencia (automático)

Los metadatos del conjunto de datos deben expresar los términos en los que este se comparte, habilitando el uso para cualquier fin y requiriendo, como máximo, la atribución.

Ver listado de licencias libres.

Formato (automático)

La extensión de archivo del recurso principal del conjunto de datos debe ser la de un formato cuya interpretación o reproducción no esté limitada por patentes o regalías.

Ver listado de formatos libres.

Criterios de calidad de datos

Validez

Cuando sea posibile, los recursos principales de cada conjunto de datos serán evaluados en su adherencia a los requisitos del formato, lo que facilita su reutilización.

  • Formatos validados
  • CSV
  • JSON
  • XML
  • KML
  • GeoJSON
